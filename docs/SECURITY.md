# 安全模型

> 本文档详细说明 HX-HouTiKu 的加密方案、威胁模型和安全建议。

---

## 概述

HX-HouTiKu 使用 **ECIES**(椭圆曲线集成加密方案)实现端到端加密。

**核心原则**: 

- 📱 私钥永远不离开你的设备
- 🔒 服务端只存密文, 即使数据库泄露也无法解密
- 📤 推送 SDK 只持有公钥, 只能加密, 不能解密
- 🔑 每条消息使用独立的临时密钥, 一条被破解不影响其他

---

## 为什么用非对称加密? 

因为代码是开源的, 如果用对称加密(共享密钥), 任何人看源码就能拿到密钥。

| 属性 | 对称加密 (AES) | 非对称加密 (ECIES) |
|------|----------------|-------------------|
| SDK 中的密钥 | 共享密钥(能加密**也能解密**) | 仅公钥(能加密, **不能解密**) |
| SDK 泄露的后果 | ❌ 所有历史消息暴露 | ✅ 攻击者只能发送新消息 |
| 代码开源的影响 | ❌ 密钥必须隐藏 | ✅ 公钥安全可公开 |
| 每条消息安全性 | ⚠️ 所有消息共用同一密钥 | ✅ 每条消息独立临时密钥 |

---

## 加密方案详解

### ECIES = ECDH + HKDF + AES-256-GCM

每条消息的加密流程: 

```
1. 生成临时 ECDH 密钥对(ephemeral_sk, ephemeral_pk)
2. ECDH(ephemeral_sk, recipient_pk) → 共享密钥
3. HKDF(共享密钥) → AES 密钥
4. AES-256-GCM(aes_key, 明文) → 密文 + 认证标签
5. 输出 = ephemeral_pk ‖ iv ‖ 密文 ‖ auth_tag
```

**前向保密**: 每条消息的临时密钥用完即丢。即使某一条消息的密钥被破解, 其他消息不受影响。

### 使用的密码学原语

| 组件 | 算法 | 说明 |
|------|------|------|
| 椭圆曲线 | secp256k1 | 比特币同款曲线, 久经考验 |
| 密钥协商 | ECDH | 安全生成共享密钥 |
| 密钥派生 | HKDF | 从共享密钥派生 AES 密钥 |
| 对称加密 | AES-256-GCM | 认证加密, 防篡改 |
| 编码 | Base64 | 密文传输格式 |

### 加密库

| 端 | 库 | 语言 |
|----|-----|------|
| 前端 PWA | [eciesjs](https://github.com/ecies/js) | TypeScript |
| Python SDK | [eciespy](https://github.com/ecies/py) | Python |

两个库互相兼容, 用 eciespy 加密的消息可以用 eciesjs 解密, 反之亦然。

## 私钥保护

用户的私钥存储在浏览器 IndexedDB 中, 经过以下方式加密保护: 

```
PBKDF2(主密码, 随机salt, 60万次迭代) → 包装密钥
AES-256-GCM(包装密钥, 私钥) → 加密后的私钥
```

### 为什么是 60 万次迭代? 

PBKDF2 的迭代次数决定了暴力破解的成本。60 万次使得: 

- 在现代 GPU 上, 每秒只能尝试约 1000 个密码
- 8 位随机密码需要约 ~200 年暴力破解
- 12 位强密码几乎不可能暴力破解

### 本地存储结构

```
IndexedDB
└── unified-push-keystore
    ├── encrypted_private_key  (AES-256-GCM 密文)
    ├── salt                   (PBKDF2 随机盐)
    ├── iv                     (AES 初始化向量)
    └── public_key             (明文, 可以安全公开)
```

## 威胁分析

| 攻击面 | 风险等级 | 防护措施 |
|--------|---------|----------|
| **源代码公开** | 🟢 低 | 非对称加密, 公钥安全可公开 |
| **Worker 被入侵** | 🟢 低 | 服务端只存密文+公钥, 无法解密 |
| **D1 数据库泄露** | 🟢 低 | 消息内容为 ECIES 密文; API Token 存储为 SHA-256 哈希 |
| **SDK 端点泄露** | 🟡 中 | SDK 只有公钥(仅能加密); 攻击者可以发垃圾消息, 但不能读 |
| **Web Push 被截获** | 🟢 低 | 推送通知只含"有新消息"信号, 不含实际内容 |
| **浏览器存储被访问** | 🟡 中 | 私钥经 PBKDF2(60万迭代)加密, 暴力破解极其困难 |
| **网络中间人攻击** | 🟢 低 | Cloudflare 强制 HTTPS (TLS 1.3); 内层还有 ECIES 加密 |
| **元数据泄露** | 🟠 已知限制 | ⚠️ 优先级和分组名是明文(设计取舍, 见下文) |

## 已知限制

以下内容**不在**端到端加密的保护范围内: 

### 1. 元数据

消息的 `priority`(优先级)和 `group`(分组名)以明文存储。

**为什么? ** 服务端需要这些信息来做路由和过滤——如果加密了, 服务端就无法按优先级推送通知、按分组查询。

**缓解措施**: 如果你的分组名本身就是敏感信息, 可以用代号代替(如用 `proj-a` 代替真实项目名)。

### 2. 流量分析

观察者可以看到"有消息发送/接收", 即使看不到内容。消息的时间戳、频率、大小都可能泄露信息。

### 3. 客户端内存

当 PWA 解锁后, 私钥存在于 JavaScript 内存中。有物理访问权限的攻击者可能通过浏览器进程提取。

**缓解措施**: 不使用时锁定 PWA(点击锁定按钮或关闭标签页)。

### 4. ADMIN_TOKEN

管理员令牌拥有完整的 API 访问权限。泄露后攻击者可以注册新接收者、发送消息(但仍然无法读取已有消息)。

---

## 安全建议

1. **使用强主密码** — 至少 12 个字符, 混合大小写、数字、符号
2. **不用时锁定 PWA** — 点击锁定按钮或直接关闭标签页
3. **定期轮换密钥** — 生成新密钥对, 重新注册, 更新 SDK 配置
4. **保管好 ADMIN_TOKEN** — 它有完整 API 权限, 不要提交到 Git
5. **使用独立 API Token** — 每个推送来源一个 Token, 权限最小化
6. **开启 2FA** — Cloudflare 账号务必开启两步验证
7. **不要在公共设备上使用** — 或者用完立即锁定/注销
